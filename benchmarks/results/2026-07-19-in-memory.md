# In-Memory TCP Baseline

Date: 2026-07-19

This is the `single_node_in_memory` baseline. A write is a 24-byte fixed binary
request (`id`, `key`, `value`) and its response is the 8-byte request ID. The
server applies the write to an in-memory hash map and immediately responds. It
does not persist, replicate, or implement Raft.

## Localhost

Server and client ran on the same macOS host using the `macos-llvm22-release`
preset (Clang 22.1.8). CPU affinity was not configured.

| Requests | Pipeline | Operations/s | p50 | p95 | p99 |
|---:|---:|---:|---:|---:|---:|
| 100,000 | 1 | 39,129.9 | 23.8 us | 38.8 us | 55.1 us |
| 1,000,000 | 64 | 378,705 | 173.2 us | 202.0 us | 231.0 us |

## Turing Pi Cross-Node

The server ran on `tpn1`; the load generator ran on `tpn4`. Both are 8-CPU
Linux `aarch64` nodes. The binaries were built with GCC 13.3 using
`-O3 -DNDEBUG -std=c++23`, and both processes were pinned with `taskset -c 0`.
The client connected to `tpn1.local`, which resolved to the switch-connected
IPv4 address `192.168.1.165`.

| Requests | Pipeline | Operations/s | p50 | p95 | p99 |
|---:|---:|---:|---:|---:|---:|
| 10,000 | 1 | 2,531.7 | 389.7 us | 401.4 us | 442.2 us |
| 1,000,000 | 64 | 59,560.5 | 1.056 ms | 1.076 ms | 1.094 ms |

The single-request case is round-trip latency bound. Pipeline 64 increases
throughput by keeping the TCP connection occupied, but this single-threaded
server responds one request at a time, so queueing raises observed latency.
An earlier Tailscale-address run is excluded because it measured the overlay
path rather than the cluster switch.

## Batched Steady State

The initial implementation issued one `send` and one `recv` syscall per frame.
The batched revision reads and responds to up to 256 frames per syscall, and
the client sends and receives frame batches while preserving one ordered
acknowledgement per `PUT`. The server was first warmed with one unreported
one-million-write run so hash-table allocation is excluded from these
steady-state measurements.

The localhost run used the release preset. The Turing Pi run used the same
`tpn1.local` path, compiler flags, and CPU pinning described above.

| Host | Requests | Pipeline | Operations/s | p50 | p95 | p99 |
|---|---:|---:|---:|---:|---:|---:|
| localhost | 1,000,000 | 64 | 2.012M | 25.9 us | 48.2 us | 85.5 us |
| localhost | 1,000,000 | 256 | 5.894M | 34.5 us | 55.0 us | 73.5 us |
| tpn4 -> tpn1 | 1,000,000 | 64 | 147,460 | 422.9 us | 428.8 us | 443.4 us |
| tpn4 -> tpn1 | 1,000,000 | 256 | 294,210 | 768.0 us | 1.157 ms | 1.172 ms |
| tpn4 -> tpn1 | 1,000,000 | 1,024 | 1.292M | 746.4 us | 924.1 us | 1.007 ms |
| tpn4 -> tpn1 | 1,000,000 | 4,096 | 1.426M | 1.352 ms | 1.530 ms | 1.599 ms |

Pipeline 4,096 is the current one-connection throughput result. Pipeline
1,024 provides nearly the same throughput with lower p50 and p99 latency, so
it is the preferred high-throughput setting until an explicit throughput-only
workload is introduced.

## Pipeline-One Latency Sweep

These runs isolate acknowledged round-trip latency. The server and client were
pinned to A76 CPU 4, and each NIC IRQ was also pinned to CPU 4. The map was
warmed before recording 20,000 pipeline-one writes. Interrupt coalescing was
the only changed setting.

| RX/TX coalescing | Operations/s | p50 | p95 | p99 | p99.9 | max |
|---|---:|---:|---:|---:|---:|---:|
| 0 us | 2,786.6 | 356.1 us | 438.7 us | 443.9 us | 912.1 us | 25.20 ms |
| 50 us | 4,147.7 | 262.2 us | 267.2 us | 275.6 us | 606.4 us | 13.33 ms |
| Driver default | 3,556.9 | 277.7 us | 287.6 us | 300.4 us | 696.8 us | 12.34 ms |

The driver rejects a 25 us setting. Zero coalescing raises interrupt pressure
enough to worsen both p50 and the long tail. The 50 us profile is the measured
latency winner. Multi-millisecond maxima are OS scheduling outliers, not cable
or switch propagation time.

## Busy-Poll Latency Profile

The RK3588 kernel supports `SO_BUSY_POLL`. For this controlled run, K3s was
stopped on `tpn1` and `tpn4`, Skiff and both NIC IRQs were pinned to A76 CPU 6,
and coalescing was set to 50 us. Each run warmed the map, then recorded 20,000
pipeline-one writes. The normal and busy-poll cases differ only in the 50 us
socket and kernel busy-poll budget.

| Profile | Operations/s | p50 | p95 | p99 | p99.9 | max |
|---|---:|---:|---:|---:|---:|---:|
| Interrupt driven | 4,122.6 | 258.1 us | 265.1 us | 269.2 us | 332.8 us | 5.27 ms |
| 50 us busy poll | 18,962.8 | 54.0 us | 56.6 us | 61.3 us | 106.8 us | 268.9 us |

Busy polling removes most interrupt wake-up delay by spending CPU time polling
for packet arrival. It is a latency profile for a dedicated core, not a
cluster-wide default. All host settings and K3s services were restored after
the measurement.

## Jumbo-Frame Probe

The direct `tpn1`/`tpn4` NICs advertise MTU 9000 capability, but the active
board-switch path did not accept it. With K3s paused and both hosts otherwise
in the CPU-6 profile, setting `end0` to MTU 9000 immediately dropped carrier.
No large-packet ping or Skiff jumbo benchmark was run. Both interfaces were
returned to MTU 1500 and carrier recovered before restarting K3s.

The isolated 1500-MTU comparison baselines were:

| Profile | Operations/s | p50 | p99 | p99.9 | max |
|---|---:|---:|---:|---:|---:|
| Default coalescing, pipeline 4,096 | 3.914M | 832.2 us | 946.5 us | 985.9 us | 43.09 ms |
| 50 us busy poll, pipeline 1 | 19,223.8 | 53.4 us | 61.0 us | 111.7 us | 228.7 us |

Do not retry MTU 9000 on this shared network until the board-switch
configuration and its supported port MTUs are known.
